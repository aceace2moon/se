# [WorkDrive] 객체지향 요구사항 분석서

---

## 1. 개요 (Introduction)
본 문서는 WorkDrive(클라우드 파일 공유 시스템)의 요구사항 정의서를 바탕으로 객체지향 분석 기법(UML)을 적용하여 시스템의 기능, 구조, 행위를 명확히 모델링하고 산출물 간의 일관성을 검증하는 데 목적이 있다.

---

## 2. 기능 모델링 (Functional Modeling)

### 2.1 유스케이스 다이어그램 액터 정의
- **사용자(User)**: 파일을 업로드, 다운로드, 공유, 검색하는 시스템의 주 사용자.
- **관리자(Admin)**: 접근 권한 통제, 보안 정책 관리 및 로그를 관측하는 주체.
- **인증 시스템(System)**: 비인가 접근을 100% 차단하기 위해 사용자의 세션과 권한을 검증하는 백엔드 요소.

### 2.2 핵심 유스케이스 설명서 (Use Case Specification)

#### [UC-01] 파일 업로드 (File Upload)
- **액터(Actor)**: 사용자(User)
- **개요**: 사용자가 로컬 파일을 시스템의 분산 저장소에 안전하게 저장한다.
- **사전 조건(Pre-condition)**: 사용자가 로그인 상태이며, 스토리지 용량이 남아 있어야 함.
- **기본 흐름(Basic Flow)**:
  1. 사용자가 업로드할 파일을 선택하여 드래그 앤 드롭한다.
  2. 시스템은 파일의 유효성 및 용량을 체크한다.
  3. 시스템은 대용량 파일의 안정적 전송을 위해 파일을 청크(Chunk) 단위로 분할한다.
  4. 시스템은 분할된 청크를 스토리지에 병렬 업로드하고 전송 상태를 사용자 화면에 표시한다.
  5. 업로드가 완료되면 성공률 메트릭을 기록하고 완료 메시지를 출력한다.
- **대안/예외 흐름(Alternative/Exception Flow)**:
  - **[예외 1] 네트워크 지연 및 단절**: 전송 중 네트워크 오류 발생 시, 시스템은 다운로드/업로드 재시도 로직을 가동하며 사용자 불편을 최소화한다.

#### [UC-02] 파일 공유 및 권한 관리 (File Sharing & Access Control)
- **액터(Actor)**: 사용자(User), 인증 시스템(System)
- **개요**: 사용자가 특정 파일에 대한 읽기/쓰기 권한을 설정하고 보안 링크를 생성한다.
- **사전 조건(Pre-condition)**: 사용자가 해당 파일의 소유자(Owner)여야 함.
- **기본 흐름(Basic Flow)**:
  1. 사용자가 공유할 파일을 선택하고 '공유 링크 생성'을 요청한다.
  2. 사용자는 대상 사용자의 접근 권한(읽기 전용 / 편집 가능)을 설정한다.
  3. 사용자는 생성된 링크의 유효 기간을 설정한다.
  4. 인증 시스템은 권한 정보를 데이터베이스에 동기화하고 토큰화된 URL을 반환한다.
- **대안/예외 흐름(Alternative/Exception Flow)**:
  - **[예외 1] 비인가 사용자의 접근**: 비인가 사용자가 링크 접근 시, 인증 시스템은 접근을 100% 차단하고 관리자 보안 로그에 위험 활동을 기록한다.

## 3. 구조 모델링 (Structural Modeling)

강의자료의 핵심 개념에 따라 주요 도메인 클래스를 식별하고 클래스의 책임과 협력 관계(CRC 카드 관점)를 정의한다.

### 3.1 주요 클래스 명세
- **User (사용자)**: 사용자의 이메일 계정 정보, 패스워드(암호화), 소유 파일 목록을 관리한다.
- **FileManager (파일 관리자)**: 파일 업로드/다운로드 중계, 청크 단위 분할 및 결합 가이드라인을 제어한다.
- **File (파일)**: 파일ID, 파일명, 용량, 저장소 경로, 버전 정보 등의 메타데이터를 가진다.
- **ShareLink (공유 링크)**: 발급된 공유 URL, 접근 권한 등급, 링크 유효 기간 정보를 관리한다.
- **AuthManager (인증 관리자)**: 세션 검증, 비인가 접근 차단(100%), 접근 권한 테이블 조회를 수행한다.

## 4. 행위 모델링 (Behavioral Modeling)

### 4.1 유스케이스 실행 시나리오 (Sequence Scenario)

#### [시나리오] 파일 업로드 및 청크 분할 전송
1. **User**가 `FileManager`에게 `uploadFile(file)` 메시지를 송신한다.
2. `FileManager`는 `AuthManager`에게 `checkSession()`을 호출하여 세션을 검증한다.
3. `AuthManager`가 유효성 결과를 반환하면, `FileManager`는 파일을 내부적으로 청크 단위로 쪼갠 후 물리 분산 스토리지에 저장을 요청한다.
4. 저장이 성공하면 `File` 객체의 메타데이터(버전 정보 포함)를 생성하고 `User`에게 업로드 성공 결과 및 상태율을 화면에 표시한다.

#### [시나리오] 비인가 접근 차단 및 관측 (Observability)
1. 익명의 사용자가 **ShareLink**를 통해 파일 접근을 요청한다.
2. `AuthManager`가 해당 요청의 토큰 및 권한을 검증한다.
3. 권한이 없는 비인가 접근으로 판명 시, `AuthManager`는 예외를 발생시키고 즉시 접근을 100% 차단한다.
4. 동시에 `AuthManager`는 로그 관리 시스템에 실시간으로 보안 위반 로그를 송신하며, 관측 가능성(Observability) 수집 지연을 5초 이하로 유지한다.

## 5. 분석 산출물 일관성 점검 (Consistency Check)

강의자료 64페이지의 지침을 기반으로 모델 간 일관성을 자가 점검하였다.

1. **기능 모델과 구조 모델의 일관성**
   - 본 분석서에 도출된 모든 도메인 클래스(`User`, `FileManager`, `File`, `ShareLink`, `AuthManager`)의 이름은 유스케이스 설명서([UC-01], [UC-02])의 흐름 시나리오 내 명사 및 주체와 정확히 일치함을 확인하였다.
2. **기능 모델과 행위 모델의 일관성**
   - 4장의 행위 시나리오에서 정의된 액터(`User`, `Admin`)는 2장의 유스케이스 다이어그램 액터와 일관성 있게 매핑되어 작동한다.
3. **구조 모델과 행위 모델의 일관성**
   - 행위 시나리오에서 객체 간 전달되는 메시지(`uploadFile`, `checkSession` 등)는 3장에서 정의된 각 클래스의 주요 연산 책임(Responsibility)의 범주 내에 존재함을 확인하였다.
